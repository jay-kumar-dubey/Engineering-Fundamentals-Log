## Q41: Friend Requests II (Medium) — LC 602
**Problem:** Find person with most total friends (requester + accepter side).

**Approach:**
- Two CTEs: UNION ALL both sides (requester, accepter) with individual counts
- Aggregate again with SUM to get total per person
- Filter with WHERE num = (SELECT MAX(num)) — guaranteed single winner

**Code:**
```sql
WITH cte AS (
    SELECT accepter_id AS id, COUNT(*) AS nums FROM RequestAccepted GROUP BY accepter_id
    UNION ALL
    SELECT requester_id AS id, COUNT(*) AS nums FROM RequestAccepted GROUP BY requester_id
),
total_cte AS (
    SELECT id, SUM(nums) AS num FROM cte GROUP BY id
)
SELECT id, num FROM total_cte
WHERE num = (SELECT MAX(num) FROM total_cte);
```

**Real-World DE Value:**
- Bidirectional relationship counting appears in graph-like data —
  social networks, supplier-buyer relationships, pipeline dependencies.
  Pattern: UNION ALL both directions → aggregate → rank.
  In DE: this is how you'd model undirected edges in a relational store
  before pushing to a graph layer (Neo4j, Neptune).

## Q42: Investment Report (Medium) — LC 585
**Problem:** Sum tiv_2016 for policyholders with shared tiv_2015 AND unique (lat, lon).

**Approach:**
- Subquery filters unique city locations: GROUP BY lat, lon HAVING COUNT(*) = 1
- Outer query filters shared tiv_2015: GROUP BY tiv_2015 HAVING COUNT(*) > 1
- Clean filter composition — no JOIN needed

**Code:**
```sql
SELECT ROUND(SUM(tiv_2016), 2) AS tiv_2016
FROM Insurance
WHERE (lat, lon) IN (
    SELECT lat, lon FROM Insurance
    GROUP BY lat, lon
    HAVING COUNT(*) = 1
)
GROUP BY tiv_2015
HAVING COUNT(*) > 1;
```

**Real-World DE Value:**
- Filtering on composite uniqueness conditions is common in data quality pipelines —
  deduplication by geo-coordinates, flagging duplicate records by composite keys.
  In DE: this pattern maps directly to Great Expectations checks and
  dbt source freshness + uniqueness tests on composite columns.
