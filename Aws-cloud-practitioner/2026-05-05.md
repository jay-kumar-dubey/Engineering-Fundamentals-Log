# AWS Log — May 5, 2026

## Topics: IAM Roles, Shared Responsibility Model

### Concepts Covered
- **IAM Roles:** Temporary permissions assigned to AWS services (not users)
  - Example: EC2 instance needs S3 access → attach a Role, not an Access Key
  - Roles are assumed temporarily — safer than hardcoding credentials
- **Service Instances + Roles:** When launching EC2, Lambda, etc. → assign role at creation
- **IAM Permissions:** Policies define allow/deny per service/action/resource
- **Security Best Practices:**
  - Never use root account for daily tasks
  - Enable MFA on root and admin users
  - Rotate access keys regularly
  - Use roles for services, not access keys

### Shared Responsibility Model
| AWS Responsible For | Customer Responsible For |
|---|---|
| Physical infrastructure | IAM users, roles, policies |
| Hardware, networking | Data encryption |
| Managed service security | OS patching (on EC2) |
| Global infrastructure | Application security |

**Core rule:** AWS secures the cloud. You secure what's in the cloud.

### Real-World Connection
- In production, every Lambda, Glue job, EC2 instance uses a Role — never hardcoded keys
- Data Engineers need to assign correct S3/Glue/Redshift permissions via roles
- Shared responsibility is a core exam concept — expect scenario questions
